# WindArms — v2 (Current Development Specification)

> **This is the primary specification — v2 is the current development target.** Gameplay implementation has not started, but **a live marketing/preview experience has** — see "What's already built" below (added 2026-07-14; this corrects the earlier blanket claim that nothing v2-related existed in code). Every new request should be assumed to target v2 unless explicitly scoped to v1. v1 is the current stable, shipped game — see [v1.md](v1.md) — kept live at `/play` and as the reference for migrating patterns (architecture, netcode, gameplay feel) into the v2 rebuild. v2 is a ground-up visual/architecture rebuild, not incremental changes on top of v1.

## What's already built (code-confirmed, 2026-07-14)

A full marketing/preview landing page occupies the site root (`src/app/page.tsx` → `LandingV2View`), replacing the v1 landing page there (v1's landing code still exists at `components/landing/LandingView.tsx` "for reference and rollback," just not currently rendered). This is presentation only — no v2 gameplay is implemented — but it's real, shipped code with real content:

- **Six sections**, data-driven (`src/components/landing/v2/sections/index.ts`): Hero, Arsenal, Operators, Skyfront, Pillars (Gameplay), Deploy (CTA). Each has a `figmaNode` slot reserved for future Figma reconciliation.
- **Named operators** — Kael Aurin and Veyra Solace — with bios, signature weapons, and accent colors. See [../gameplay/operators.md](../gameplay/operators.md) (rewritten 2026-07-14 to make this the canonical source, superseding the original generic four-archetype brief).
- **A named wind-weapon arsenal** — Aeolus Rifle, Vortex Carbine, Tempest Cannon, Gust Blade. See [../gameplay/weapons.md](../gameplay/weapons.md#v2-arsenal-windweaponsts).
- **A named world** — "The Skyfront," a megacity of floating islands with points of interest (Wind Temple, Storm Reactor, Sky Bridges, Airship Docks). See [../design/skyfront.md](../design/skyfront.md).
- **A real color palette** — the "STORM" design tokens (`src/lib/v2/tokens.ts`), mirrored 1:1 into Tailwind and consumed by Three.js materials. See [../design/art-direction.md](../design/art-direction.md#storm-design-tokens-implementation-accurate).
- **A hybrid asset pipeline** — procedural-first, with a resolver (`src/lib/v2/assetResolver.ts`) that probes `public/v2-art/` for real art and falls back gracefully when none exists. See [../technical/asset-pipeline.md](../technical/asset-pipeline.md).

This decision — treating the coded preview as canon over the original text brief where they diverge — is logged in [decisions.md](../decisions.md).

## Relationship to v1

v2 does not modify v1's gameplay code or behavior — the playable game at `/play` is untouched. The v1 build (all phases, engineering notes, deployment) stays exactly as documented in [v1.md](v1.md) unless and until v2 *gameplay* implementation work explicitly begins; the v2 *landing/marketing* layer is a separate, already-started track that doesn't touch `/play`. Planned/undecided items that don't yet have a home in v1 or the v2 design docs above are tracked in [roadmap.md](../roadmap.md).
