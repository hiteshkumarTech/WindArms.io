## 2026-07-18

- **Skyfront Trial polish pass: corrected FP weapon pose, barrel-origin muzzle/tracer, and a Low/Medium/Max difficulty system.** No new GLBs, no GLB modifications, no operators/arms, no multiplayer.
  - **FP pose fix.** New `src/lib/v2/weapons/vortexViewmodelPose.ts` (`VORTEX_VIEWMODEL_POSES` — typed hip/ADS `{position, rotation, scale}`) replaces `VortexViewmodel.tsx`'s old scattered `REST`/`ADS_REST`/`VIEWMODEL_SCALE` constants. Root cause: the loaded GLB's local +X (muzzle-forward) axis was never rotated to align with camera-forward — only translated into view, never corrected in rotation. Fixed with a ground-truth-verified `rotateY(+π/2)` correction (isolated zero-rotation screenshot test first, then the correction alone, then the natural tilts layered back on). Shared unchanged by `/v2/range` and `/v2/play`. Reasoning + before/after: `docs/decisions.md`.
  - **Barrel-origin muzzle/tracer.** New `src/lib/v2/weapons/vortexRuntimeAnchors.ts` (`VORTEX_RUNTIME_ANCHORS.muzzleLocal`) — a TEMPORARY hand-measured local coordinate near the barrel's +X tip, since the real GLB has no authored `socket_muzzle` yet (Blender export still v0.2). New `src/lib/v2/range/muzzleWorldPose.ts` — a per-frame world-space position/direction singleton bridge, published every frame by `VortexViewmodel` (`localToWorld` after a forced `updateWorldMatrix`) and consumed by `VortexFireSystem` for the *visual* tracer/muzzle-flash origin only. The gameplay aim ray (`raycaster.set(camera.position, dir)`, hit detection, damage, spread, recoil) is untouched and stays 100% camera-based. Verified visually across idle/moving hip-fire, sprint, reload, inspect, and sustained ADS-fire (live tracer now originates at the barrel tip, not the receiver).
  - **Difficulty system.** New `src/lib/v2/play/difficulty.ts` — `TrialDifficulty`, `TrialDifficultyConfig`, `TRIAL_DIFFICULTIES` (Low/Medium/Max), `resolveDroneConfig()`, `resolveDroneSpawns()`. Medium's multipliers are all `1` and its drone count/match time equal the pre-existing `TRIAL`/`DRONE` constants exactly — the original tuning was not rebalanced. `matchStore.ts` gained `selectedDifficulty`/`selectDifficulty()` (locked once countdown begins) plus `useSelectedDifficulty`/`useDifficultyConfig`/`useTotalDrones` selectors; `beginCountdown()`/`restart()` both re-bump `restartNonce` so drones re-seed with the locked-in preset right as combat starts. `DroneEnemy.tsx`, `DroneSquad.tsx`, `DroneBoltPool.tsx` now resolve HP/damage/fire-cadence/accuracy/projectile-speed/movement-speed through `resolveDroneConfig` instead of the raw base constants; `DroneSquad`'s spawn list is difficulty-reactive (`resolveDroneSpawns`). `V2PlayView.tsx`'s pre-countdown overlay gained a three-card difficulty selector (default Medium, "DEPLOY — {DIFFICULTY}" button); `V2PlayHud.tsx`, `MatchOverlay.tsx` and `EndMatchScreen.tsx` show the effective drone total and a difficulty badge, all resolved through the same store selectors. Player health/movement and the Vortex Rifle's own damage output/recoil/fire-rate are untouched by any preset; what difficulty *does* change is the damage the player takes (`droneDamageMultiplier` scales drone bolt damage, an enemy stat). Verified via Playwright: pre-deploy Low→Max→Low switching, Low/Max HUD correctness (5/3:30 and 8/2:45 respectively), Restart-preserves-difficulty, fresh-route-entry-resets-to-Medium — zero console errors across all of it. Full design + the multiplier table: `docs/gameplay/skyfront-trial.md`.
  - Validation: `typecheck`/`lint`/`build` all clean; `/`, `/play`, `/v2/range`, `/v2/play` all load with no new console errors/404s (landing's pre-existing marketing-image 404s and V1's pre-existing SSAO warning are unrelated to this pass — neither route's files were touched). LOD routing, V1, and the landing showpiece transform are unaffected.

Extracted `TargetUserData` → `src/lib/v2/combat/targets.ts` (shared by range targets + drones); moved `VortexFireSystem` → `three/weapons/` (range path now a shim) with an optional match `combatGateRef`. Landing V2 Play buttons (hero, CTA, footer, navbar) now open `/v2/play`. V1 `/play`, `/v2/range`, and the landing LOD0 showpiece untouched. Full design: `docs/gameplay/skyfront-trial.md`.
- **Vortex Rifle v0.2 source accepted** (`Hitem3d-1784224974921.glb` → `WindArms Assets/Weapons/VortexRifle/vortex_v0.2_source.glb`): ~1.99M tris / 87.5 MB high-poly, realistic carbine proportions (1.00 × 0.27 × 0.14 m), supersedes v0.1 as the bake source. Inspection report: `docs/forge/vortex-rifle-v0.2.md`; v0.1 report annotated as superseded. No modification to the asset.
- **Vortex Rifle v0.2 runtime derivative shipped + visual integration pass.** New `tools/make-vortex-runtime.mjs` (gltf-transform v4 programmatic: dedup/weld/meshopt-simplify, +90° Y orientation bake → muzzle +X, WebP ≤2048, Draco, per-LOD fresh reads, output backup, budget gates). Verified outputs: `public/v2-art/vortex-rifle.glb` LOD0 139,598 tris / 80,289 verts / 0.84 MB (2 WebP textures, 0.35 MB) and `vortex-rifle.lod1.glb` LOD1 55,834 tris / 33,573 verts / 0.57 MB — both pass their budgets; real mesh confirmed rendering on the landing hero (`realVortexNode=VortexRifle_LOD0`, fallback no longer shown). Fixed `tools/inspect-glb.mjs` world-bounds bug (single-root-scale shortcut ignored rotations → reported pre-rotation axes; now full node-transform accumulation — see decisions.md). Hero composition fixed via documented DISPLAY scale 2.9 (vs physical 0.68) in `visualConfigs.ts` with proportionally rescaled rim/glow (see decisions.md). Known limits recorded in `docs/forge/vortex-rifle-v0.2.md`: automatic decimation (not retopology), single `pbr_material` (accent tint inert), no normal map.
- **Vortex Rifle LOD routing + validation budget cleanup.** `/v2/range`'s `VortexViewmodel` now requests LOD1 (55,834 tris) explicitly via a new `PipelineModel`/`useResolveModelSlot` `requestedLod` override, independent of the landing hero's quality-driven LOD0 default — same slot, no duplicated asset entry. Fixed `manifest.ts`'s `vortex-rifle.budget` (was the stale 18,000-triangle `__template` placeholder, never updated once real numbers existed) via a new per-LOD `budgetByLod` field, checked against whichever tier actually resolved. A separate `requestedLod`-versus-resolved-LOD check warns about accidental tier fallback. Emptied `requiredSockets`/`requiredClips` (true today — nothing currently reads either) and moved the v1.0 target set to new informational `plannedSockets`/`plannedClips` fields. Temporary `PipelineDebugProbe` (render proof) removed after confirming both tiers load correctly — see docs/decisions.md.
- **Phase 5 — operator architecture** (production foundation for Operator 01 — Kael Aurin; no GLB yet, no gameplay, V1 untouched). New: `src/lib/v2/operators/` (OperatorDefinition/Metadata/VisualConfig/AnimationConfig/AttachmentConfig/SkinConfig; 16 animation states; 16 typed sockets with humanoid bone fallbacks; slot naming; kael+veyra registry referencing the canon roster in `content/operators.ts`), `src/components/three/operators/` (OperatorModel + per-instance skeleton cloning, render modes full/armsOnly/bodyHidden/shadowOnly, FirstPersonOperatorRig sway/recoil pivots, OperatorPawn TP contexts, OperatorSocketAnchor, procedural OperatorSilhouette fallback, animation/socket hooks), `src/components/v2/operators/OperatorShowcase.tsx` (hero lighting + turntable + camera/background extension points), `tools/inspect-operator.mjs` (character GLB gate: skeleton/clips/sockets/LOD budgets, CI exit codes). Extended (additive): `pipeline/types.ts` SocketName/ClipName unions, `pipeline/manifest.ts` operator entries + shared `OPERATOR_BUDGET`. Report: `docs/forge/operator-pipeline.md`.
