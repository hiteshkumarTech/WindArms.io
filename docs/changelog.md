- **Milestone 7, Phase F, Step 6E — root cause of "arms invisible in REST MESH DIAGNOSTIC" finally identified: not a transform bug, a viewing-angle/pose-appropriateness mismatch. Two independent latent bugs also found and fixed along the way (debug-helper matrixWorld staleness; a from-scratch camera-parenting diagnostic that would itself have been silently invisible).** Manual browser testing of Step 6D's fixes still showed the arms invisible and the bounding box detached/oversized with REST MESH DIAGNOSTIC + bounding box on. Built the full isolation toolkit the brief requested rather than theorizing further.
  - **New `classifyVerticesInCameraSpace()` (`kaelArmSolve.ts`)** — computes true camera-space/NDC classification (front/behind, near/far-plane clipping, in-frustum vertex count, NDC min/max, camera-space center) for every deformed vertex of a SkinnedMesh, using the same view/projection math the GPU rasterizer uses. Answers "is this geometry actually on screen" independently of any bounding-box math. 6 new tests (in-frustum, behind-camera, near-clip, outside-horizontal-FOV, far-clip, finite/consistent center).
  - **ROOT CAUSE, found by running this new classifier headlessly against the real rebuilt asset at every stage of the transform chain:** with the shipping `shoulderRootOffset = [0, -0.15, -0.3]`, the arm's REST POSE (hanging naturally down from the shoulder, exactly as authored for third-person viewing) sits almost entirely BELOW the visible frustum from this close a first-person camera — only 0.1% of vertices (6 of 10,512) land inside frame; NDC Y spans `[-3.85, -0.15]`, i.e. from far below the bottom edge (-1) to just barely touching it. This is NOT a coordinate-space bug, NOT a double transform, NOT a stale matrix, NOT a material/culling/layer issue (all explicitly tested and disproven below) — it is a natural consequence of showing a *rest* pose (arms dangling at the character's sides) from point-blank shoulder range, a viewing angle a rest pose was never meant to be seen from. The SAME headless sweep at IK weight 0.25/0.5/1.0 (the actual shipping behavior) shows the arm progressively entering frame as the hand is pulled toward the grip target: 0.1% → 36.3% → 66.7% in-frustum — correct, healthy, expected behavior. Conclusion: the underlying transform/recenter/skinning/IK chain was already correct; REST MESH DIAGNOSTIC was never going to look like "two visible arms" by design, and the Step 6D/6E acceptance criteria asking for that was based on a mistaken assumption. The real acceptance check going forward is IK-enabled (shipping) mode with the bounding box on, not the rest-pose diagnostic.
  - **Confirmed real bug #1 (debug-only, does not affect the actual rendered frame): bone `matrixWorld` staleness for same-tick debug reads.** `restoreRestPose`/`solveSide` mutate bone LOCAL transforms; the existing `container.updateWorldMatrix()` call each frame happens BEFORE that mutation, so any code reading raw `matrixWorld` later in the SAME tick (the bounding-box helper's separate `useFrame`) saw the PREVIOUS frame's pose, not the just-restored/just-solved one. The actual GPU draw was never affected (three.js's renderer refreshes the whole scene's matrices right before rasterizing, after all `useFrame` callbacks finish) — this only ever corrupted the debug overlay. Fixed with an explicit `container.updateWorldMatrix(true, true)` (or `instance.updateWorldMatrix(true, true)` for the camera-parented diagnostic path) immediately after every bone-mutating call. New regression test proves the staleness window is real and the fix closes it.
  - **Confirmed real bug #2, caught before shipping: a from-scratch diagnostic mode would itself have been silently invisible.** Building the brief's requested "DIRECT CAMERA MOUNT" (parent the arms directly onto the camera object) surfaced a genuine three.js gotcha: `WebGLRenderer.render()` builds its render list by traversing from `scene`, never from `camera`, and R3F's default camera (as configured by `RangeScene.tsx`'s `camera={{...}}` shorthand) is NOT itself added to the scene graph. Anything parented directly to such a camera would never be visited by that traversal — invisible by omission, regardless of material/visible/frustumCulled. Fixed by explicitly adding the camera to the scene (`scene.add(camera)`, idempotent, harmless since `scene`'s own transform is identity) before parenting the diagnostic group. Confirmed via `node_modules/three`'s own renderer source (`camera.parent === null` is explicitly the expected/common case; `projectObject(scene, camera, ...)` starts from `scene`).
  - **Built "DIRECT CAMERA MOUNT"** — the hardest isolation diagnostic requested: the complete clone (mesh + full bone hierarchy, moved together, never the mesh alone) parented directly onto the camera via true THREE.js parent-child, bypassing the ENTIRE per-frame transform chain (camera-copy, `shoulderRootOffset`, IK, wrist correction, finger posing), with every material swapped for a depth-ignoring/double-sided/fully-opaque override and `frustumCulled` disabled. New `?ik=1` tuner button. Deliberately keeps the recenter anchor applied (a fixed asset-space correction, not "runtime logic under test" — disabling it would just reproduce the already-diagnosed, already-fixed "mesh 1.5m above camera" symptom). Local offset `[0, 0.2, -0.6]` was chosen by headlessly sweeping candidate offsets against the real asset (not the brief's placeholder `[0,-0.2,-0.8]`, which only gets 4.5% of vertices in frame) — this offset gets 100% in-frustum at a comfortable ~0.58m.
  - **Hypotheses tested and DISPROVEN by direct measurement, not assumption:** (1) backface culling from inverted winding — material is `THREE.DoubleSide`, confirmed by direct inspection, so winding cannot hide anything; (2) camera render layers — grepped the entire `src/components/three` tree, no custom layers used anywhere, camera and mesh both sit on the default layer; (3) a second/dedicated viewmodel camera (a common FPS "avoid near-clip" trick) — `VortexViewmodel.tsx` has zero camera/FOV/near/far references, confirming the rifle (and by extension the arms) use the one single main camera; (4) the debug bounding-box helper double-applying the camera transform — read `Box3Helper`'s three.js source directly: it sets its OWN local position/scale from the box's world-space center/size, and it's mounted as a direct `<Canvas>` child (identity parent) in `RangeScene.tsx`, so this is mathematically correct as-is, no double transform.
  - Full investigation, every measurement, and the exact staged-transform data (recenter-only / +shoulderOffset / +IK at each weight): `docs/decisions.md`.

Investigated headlessly rather than guessing further.
  - **Confirmed real bug: the bounding-box debug helper never reflected the actual deformed pose.** `ArmBoundingBoxHelper` used `THREE.Box3.setFromObject()`, which uses each mesh's STATIC `geometry.boundingBox` (bind-pose, computed once at load) transformed by the mesh's current `matrixWorld` — it does not account for GPU skinning at all (skinning happens in the vertex shader; the CPU-side geometry attributes never change). Proved via a headless diagnostic against the real asset: `Box3.setFromObject` returned the IDENTICAL box at IK weight=0 and weight=1, and that box sat ~1.5m away (in Y) from where the TRUE deformed geometry (measured via `SkinnedMesh.applyBoneTransform`, the same math the GPU shader runs) actually renders. Fixed with a new `computeDeformedSkinnedBounds()` (`kaelArmSolve.ts`) that walks every vertex through the real skinning transform; verified with a new synthetic-SkinnedMesh test suite (rotating a bone visibly moves the computed bounds; the old static method would not have).
  - **Built a true rest-pose bypass as a harder ground-truth diagnostic than "IK disabled."** The existing `ikDisabled` toggle still ran the IK solver with `weight=0`, exercising `solveTwoBoneIk`'s weight-blend math — mathematically equivalent to rest pose, but not a hard guarantee. - **Milestone 7, Phase F, Step 6C — `operator-kael-arms.glb` rebuilt: root cause of "exploded geometry" confirmed as a decimation artifact at the open shoulder cut, not the extraction threshold or the IK/skinning runtime.** Manual browser testing after the shoulder-offset change (below) still showed giant polygon slabs, correctly ruling out camera-distance tuning as a fix. Investigated with real Blender automation (`E:\Program Files\Blender Foundation\Blender 5.2\blender.exe`, headless) against the accepted rigged source (`kael_v0.1_source.glb`, 162,155 verts) rather than guessing: measured boundary-edge lengths at every stage of the existing `tools/blender/make-kael-fp-arms.py` pipeline. Pre-decimation extraction boundary edges are tiny everywhere tested (max ~0.017m, median ~0.0035m, consistent across thresholds 0.15–0.65) — the raw weight-threshold cut is NOT the defect. Post-decimation, the SAME mesh's boundary edges reached 0.121m (7x) and max face area grew 104x (0.000082 → 0.008560 m²) — the Decimate modifier's Collapse operation, applied with zero protection at the open shoulder/torso boundary (only hand/finger vertices are protected), stretches triangles there with no far-side neighbor information to constrain it.
  - **Two alternative fixes were tried and rejected, measured not assumed.** (1) Protecting boundary-adjacent vertex rings from decimation (up to 5 rings) — measured to NOT fix it (edges still reached 0.135m) and blew the triangle budget; forcing decimation away from a protected region just relocates the same distortion to the boundary of that region (a "waterbed effect"), it doesn't remove it. (2) Capping the open boundary with `bmesh.ops.holes_fill` before decimating (including from a clean planar bisect through the upper-arm bone's own rest axis, not just the irregular weight-threshold boundary) — measured to fail outright: only 2–4 new faces were created from 1,700–8,000 boundary edges, leaving most of the boundary open and introducing new malformed/wire topology that hadn't existed before. The real boundary topology is too irregular for automatic hole-filling on this mesh.
  - **The fix that measured clean: a post-decimation "scalpel."** New `MAX_EDGE_LENGTH_M = 0.035` cleanup step in `make-kael-fp-arms.py`'s `final` mode — after decimation, delete any face whose longest edge exceeds this length, then remove the resulting loose vertices. On the real asset (threshold=0.35, target=20,000 tris — the values that built the previously-shipped file): 276 of 19,996 triangles (1.4%) exceeded the threshold; removing them dropped max boundary edge to 0.034m (from 0.121m, 4.7x) and max face area to 0.000318 m² (from 0.008560, a **27x reduction**) — full comparison against the original file's actual geometry (not just the isolated test): max edge 0.1615m → 0.0344m, max face area 0.008674 → 0.000318 m². Triangle count barely moved (19,996 → 19,720).
