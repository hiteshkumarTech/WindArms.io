**Steps taken:**
- Removed non-character objects (none were present besides the armature+mesh — confirmed, not just assumed).
- Removed the imported `Armature|mixamo.com|Layer0` action — Mixamo's default embedded clip, not one of this project's 16 authored animation states, matches the source brief's "no junk clips" rule.

## Facing-direction — empirically proven, then corrected

`tools/blender/verify-kael-facing.py` — imports the **already-exported GLB** (not the FBX, so this proves what actually ships), places two orthographic cameras at world +Z and world −Z (glTF/runtime convention — Three.js's forward, not Blender's up axis), both looking at the character, and renders labeled PNGs. Axis mapping derived from Blender's own `export_yup=True` inverse conversion (`gltf_Z = -blender_Y`), spelled out in the script's header rather than asserted from memory.

**First run (pre-correction):**
| Render | Shows |
|---|---|
| `kael-view-from-plus-z.png` | **Face** — eyes, nose, mouth, chest emblem all visible |
| `kael-view-from-minus-z.png` | **Back of head/torso** |

**Result: Kael's visual forward = +Z. WindArms required forward = −Z. Correction required = yes.** This confirms the earlier math-based hypothesis, but as an actual observation of rendered pixels, not a repeat of the same reasoning.

**Correction applied** in `tools/blender/make-kael-source-glb.py`: a fixed 180° rotation around Blender's up axis (Z), applied to the armature and mesh **together** via the same `transform_apply` mechanism already used for the scale/rotation cleanup — never mesh-only, which would desync the armature modifier's bind reference from vertex data. GLB regenerated.

**Second run (post-correction), same script, same two camera positions:**
| Render | Shows |
|---|---|
| `kael-view-from-plus-z.png` | Back of head/torso |
| `kael-view-from-minus-z.png` | **Face** |

Inverted exactly as expected. Kael now faces **−Z**, matching the WindArms convention. Height (1.8300 m) and grounding (min-Z 0.000000) were re-measured after correction and are unchanged — a pure yaw, no distortion.

(Minor, non-blocking: the procedural floor/grid the verification script also adds didn't render visibly in either frame, likely camera framing. Doesn't affect the facing determination, which rests entirely on the character's own face-vs-back geometry.)

**Budget check — the tool's 2 hard errors are expected, not a rejection.** `inspect-operator.mjs` reports triangles (299,915 / 45,000) and file size (11.50 MB / 6 MB) as `FAIL` against the **runtime LOD0 budget**. It is not evidence against acceptance; it's confirmation of exactly how much decimation work `tools/make-kael-runtime.mjs` (Phase F, not yet built) has ahead of it — proportionally similar in scale to the Vortex Rifle's own ~93% reduction.

## Final classification: **ACCEPTED WITH LIMITATIONS**

Structurally, this rig is clean: 65 bones, single root, all required limb chains, all 10 finger chains, 100% weighted vertices, a passing deformation smoke test on all 4 sampled bones, and correct 1.83 m normalization with feet grounded at the origin. Forward direction is now empirically verified and corrected (see above) rather than left as an open question.

But **visual integration is not complete**, which is why this is not a plain ACCEPTED:

1. **No texture data.** Confirmed two independent ways — Blender's own `has_data: False` check, and the render itself showing flat missing-texture magenta. The character will render untextured until real texture data is sourced. Not a rig defect; a material/asset gap.
2. **Source geometry is high-poly by design** (299,915 tris) and requires the not-yet-built runtime derivative (`tools/make-kael-runtime.mjs`) before it's usable at any LOD budget — expected, not a regression, but real remaining work.

**This source is valid for pipeline and IK development, but not final visual presentation.** Do not treat the untextured asset as visually production-ready.
